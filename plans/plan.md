# ContextWrap 规划

## 核心定位

ContextWrap 不应一开始就被定义成“复制大段代码上下文”的插件。

更准确的产品定义应该是两层能力：

1. **坐标引用模式**
   - 快速把“当前代码位置”编码成 AI 和人都容易理解的引用格式。
   - 目标是节省 token，让你在提示词里用最小信息准确指向代码位置。
   - 剪贴板里优先放“文件 + 行号 / 行号范围”，而不是源码正文。

2. **上下文复制模式**
   - 当你真的需要把代码发给别人，或者直接把上下文贴给 AI 时，再复制实际代码块。
   - 这是第二阶段能力，建立在前面的“坐标定位”和“范围选择”之上。

换句话说，**MVP 先做“引用代码位置”，再做“复制代码内容”。**

## 第一阶段：坐标引用 MVP

这一阶段尽量保持语言无关，因为它本质上只处理 buffer、文件名和行号。

### 1. 当前行定位

在普通模式下，插件把当前光标所在位置复制成结构化引用。

建议默认格式：

- `@relative/path/to/file.py:L42`

可选模板化格式：

- `@relative/path/to/file.py#L42`
- `file=relative/path/to/file.py line=42`
- `请查看 @relative/path/to/file.py:L42`

### 2. Visual 模式范围定位

在 Visual 模式下，插件复制选区对应的行号范围。

建议默认格式：

- `@relative/path/to/file.py:L42-L57`

如果是单行选区，可以退化成：

- `@relative/path/to/file.py:L42`

### 3. 统一的结构化模型

内部不要直接拼字符串，而是先构建统一数据结构，再交给模板层渲染。

建议内部模型：

```lua
{
  path = "relative/path/to/file.py",
  start_line = 42,
  end_line = 57,
  mode = "current_line" | "visual_range" | "context_range",
  language = "python",
}
```

这样以后可以很方便支持：

- 不同剪贴板模板
- 给 AI 的 prompt 模板
- 给人看的简洁格式
- 后续“按坐标再去取代码”的二阶段能力

### 4. 模板化输出

这一部分很重要，因为你的真实需求不是“复制坐标”本身，而是**快速得到一段可直接贴进提示词的引用**。

建议支持至少两类模板：

- `reference`
  - 只输出坐标
- `prompt`
  - 输出一段可直接发给 AI 的文本

例如：

```text
请结合以下代码位置分析问题：
@relative/path/to/file.py:L42-L57
```

或者：

```text
Focus on @relative/path/to/file.py:L42
```

## 第二阶段：坐标版上下文选择

这一阶段开始引入“上下文”概念，但**剪贴板里仍然只放坐标，不放源码正文**。

目标是：用户不用手动进 Visual 选区，插件自动帮用户推断“当前最值得引用的代码范围”。

### 1. 典型选择目标

- 当前语句
- 当前赋值语句
- 当前 import 语句
- 当前函数
- 当前方法
- 当前类
- 当前控制块
- 当前模块级定义

输出仍然类似：

- `@foo/bar.py:L10-L28`

### 2. 为什么这一层很关键

这层实际上是“AI 定位优化器”。

你在和 AI 对话时，很多时候并不需要完整代码，只需要告诉它：

- 看这个函数
- 看这个类
- 看这个 import
- 看这个 if/try/with 代码块

所以它的核心价值是：

- 比“只给当前行”更准确
- 比“直接复制整个代码块正文”更省 token
- 更适合模板化提示词

### 3. 这一层需要处理的情况

对于“自动选择上下文范围”，插件会遇到这些情况：

#### A. 光标语义位置

- 光标落在普通语句里
- 光标落在赋值语句右侧表达式
- 光标落在函数名、参数、调用表达式
- 光标落在类体内部
- 光标落在 import 语句

#### B. 结构边界

- 单行语句
- 多行语句
- 多行函数签名
- 带 decorator 的函数或类
- 含 docstring 的函数或类
- 链式调用 / 长表达式跨多行

#### C. 嵌套结构

- 函数内函数
- 类内方法
- if / for / while / with / try / except / match
- 模块级定义中再嵌套子块

#### D. 范围优先级

找到候选范围后，还要决定复制哪个层级：

- 当前语句
- 包裹当前语句的函数
- 包裹当前函数的类
- 包裹当前类的模块级范围

所以这里一定要有“就近优先”或“指定层级”的策略。

## 第三阶段：实际代码上下文复制

这是最初你提到的那类能力：真正把源码复制出来。

这一阶段建立在前两阶段之上：

- 先定位出一个坐标范围
- 再读取对应源码
- 再按模板输出

适用场景：

- 把代码上下文发给别人
- 直接把代码片段贴给 AI
- 附带文件名和行号一起输出

建议输出格式：

```text
# relative/path/to/file.py:L42-L57
<actual code here>
```

或者：

```text
File: relative/path/to/file.py
Lines: 42-57

<actual code here>
```

## Python 在这个计划中的位置

现在 Python 最适合放在**第二阶段和第三阶段**。

原因是：

- 第一阶段只需要文件和行号，几乎不依赖语言语义。
- 第二阶段开始需要识别“当前函数 / 类 / 语句 / 控制块”的边界，这时 Python 语法就变重要了。
- 第三阶段如果要做更聪明的“上下文扩展”，Python fixture 也会成为主要验证基础。

所以现在更合理的策略是：

- **阶段 1：语言无关**
- **阶段 2：Python first**
- **阶段 3：Python first，再考虑泛化**

## 测试语料设计

测试目录仍然建议保留在仓库内，方便后续回归：

- `tests/`
  - `languages/`
    - `python/`
      - `fixtures/`
      - `cases.lua` 或 `cases.json`

### 测试集合分层

建议分成三类：

1. **定位测试**
   - 当前行
   - Visual 选区
   - 相对路径 / 绝对路径处理
   - 单行 / 多行范围格式

2. **上下文范围选择测试**
   - 函数
   - 类
   - import
   - 赋值语句
   - 控制块
   - 嵌套结构

3. **代码复制测试**
   - 坐标转源码
   - 是否带头部元信息
   - 多行截取边界是否正确

### Python fixture 优先覆盖的场景

- 顶层变量 / 常量
- import / from import / alias import
- 普通函数
- decorator 函数
- class 与 method
- nested function
- if / for / while / with / try / except / match
- 多行表达式
- docstring
- 跨文件 import

## 建议的命令形态

命令名后续可以再斟酌，但能力上建议拆成这四个：

- `:ContextWrapHere`
  - 复制当前行坐标

- Visual 模式下 `:ContextWrapRange`
  - 复制选区范围坐标

- `:ContextWrapContext`
  - 自动选择当前上下文范围，但只复制坐标

- `:ContextWrapCode`
  - 复制实际代码上下文

如果你想进一步节省心智负担，也可以做成一个命令配多个 mode：

- `:ContextWrap here`
- `:ContextWrap range`
- `:ContextWrap context`
- `:ContextWrap code`

## 结论

新的产品顺序应该是：

1. **先解决“我怎样最低成本告诉 AI 我在看哪段代码”**
2. **再解决“我怎样让插件自动帮我选这段代码的范围”**
3. **最后才是“我怎样把实际代码内容一并复制出来”**

这个顺序比一开始直接做“智能复制上下文代码”更稳，也更贴近你真正的使用场景。
